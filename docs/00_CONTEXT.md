# Контекст, цели и границы

## Исходная ситуация

В инфраструктуре используется OpenStack Epoxy 2025.1. Развертывание и обновление выполняются через Kolla Build и Kolla-Ansible на Rocky Linux. В control plane уже работают MariaDB, RabbitMQ, etcd, Mistral, Watcher и Masakari.

Horizon предоставляет широкий набор панелей и плагинов, но не подходит как единственная основа для интерфейса, который должен быстро обрабатывать крупный inventory и часто получать специализированные функции. Skyline быстрее как базовый UI, но его расширение не покрывает требуемую прикладную модель. Поэтому целевое направление — собственный портал, который сосуществует с Horizon в переходный период и использует только официальные API OpenStack.

## Пользовательская цель

Оператор должен из одного интерфейса:

- находить и анализировать виртуальные машины и гипервизоры без загрузки всего inventory в браузер;
- видеть связанный проект, host, availability zone, flavor, состояние, alarms и историю операций;
- создавать логические группы ВМ и хостов, включая явное и динамическое членство;
- выполнять только разрешенные его роли действия;
- запускать утвержденные многошаговые операции и наблюдать их прогресс;
- анализировать прикладной аудит и переходить по correlation ID к внешнему аудиту;
- получать частичный результат, если один из регионов или сервисов временно недоступен.

Администратор продукта должен добавлять новый OpenStack API, новый вид ресурса или workflow как отдельный адаптер и модуль, не переписывая shell приложения и модель безопасности.

## Техническая цель

Создать BFF-портал с собственной read model, каталогом capabilities и workflow, server-side сессиями и наблюдаемыми асинхронными операциями.

Портал не является новым identity provider, новым OpenStack policy engine, полноценным SIEM, Vault, backup-системой или заменой Mistral.

## Границы ответственности

### Портал отвечает за

- представление данных;
- прикладные группы и метаданные;
- прикладные роли и capabilities, сужающие OpenStack-права;
- безопасную серверную сессию;
- нормализацию API;
- запуск разрешенных workflow;
- статус операций;
- аудит собственных действий;
- экспорт событий во внешний централизованный аудит;
- документацию собственных API и функций.

### OpenStack отвечает за

- идентификацию и токены Keystone;
- окончательную авторизацию OpenStack API через service policies;
- lifecycle ресурсов;
- планирование и размещение;
- Mistral workflow execution;
- Watcher actions;
- Masakari recovery;
- service notifications.

### Внешний корпоративный контур отвечает за

- IAM/IdP, MFA и segregation of duties;
- корпоративный УЦ, сертификаты и их выпуск;
- SIEM и неизменяемое долговременное хранение аудита;
- Vault (SecMan) и ротацию инфраструктурных секретов;
- PAM, sudo, session recording и auditd на узлах;
- backup и RBAC на backup-хранилище;
- СХД и запрет local file-backed VM disks;
- network segmentation и firewall;
- утверждение исключений из конфликтующих требований ДКБ.

## Модель поставки

### Уровень P0: локальный функциональный PoC

Работает на mock OpenStack и локальных MariaDB/RabbitMQ. Доказывает архитектуру, таблицы, группы и workflow UI. Не является доказательством соответствия ДКБ.

### Уровень P1: интегрированный read-only PoC

Работает с тестовым OpenStack через реальную федеративную идентификацию или эквивалентный тестовый контур, HTTPS, server-side сессии, RBAC и аудит. Не выполняет опасные изменения инфраструктуры.

### Уровень P2: интегрированный mutating PoC

Запускает allowlisted Mistral workflow в отдельном тестовом проекте, имеет idempotency, operation tracking, отрицательные тесты авторизации и доставку аудита.

### Уровень P3: production pilot

Развернут через Kolla, использует корпоративные PKI, SIEM, Vault (SecMan), registry, сетевую сегментацию и HA. Имеет протокол нагрузки, failover и формальный gap analysis ДКБ.

## Non-goals PoC

- Полная функциональная замена Horizon.
- Поддержка всех сервисов OpenStack в первой версии.
- Прямое управление гостевой ОС.
- Прямое чтение service database OpenStack.
- Выполнение произвольных скриптов из UI.
- Создание собственного универсального workflow engine.
- Доказательство полного аудита действий root, DBA, storage admin и пользователей гостевых ОС.
- Формальная сертификация соответствия ДКБ.

## Критические неизвестные для E00

- Реальное число ВМ, гипервизоров, регионов, проектов и одновременных пользователей.
- Используемый IdP и способ federation.
- Доступный API SIEM и требования к сроку хранения.
- API и модель доступа корпоративного Vault (SecMan).
- Включенные OpenStack notifications и отдельный notification transport.
- Политика storage для ephemeral disks и snapshots.
- Базовый образ Kolla, container runtime и корпоративный registry.
- Перечень разрешенных прикладных ролей и workflow.
