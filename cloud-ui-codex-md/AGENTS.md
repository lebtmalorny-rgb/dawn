# Инструкции для Codex

## Назначение репозитория

Репозиторий реализует расширяемый административный портал для OpenStack Epoxy 2025.1. Целевая среда использует Kolla Build, Kolla-Ansible и Rocky Linux; в кластере уже присутствуют MariaDB, RabbitMQ, Mistral, Watcher, Masakari и etcd.

Работа считается успешной только тогда, когда появляется наблюдаемое поведение, тесты и доказательства выполнения требований. Генерация файлов без запуска проверок не считается завершением задачи.

## Порядок начала любой задачи

1. Прочитай этот файл полностью.
2. Прочитай файл текущего этапа в `tasks/`.
3. Прочитай только перечисленные в нем документы из `docs/`.
4. Осмотри фактическое состояние репозитория. Не предполагай, что структура или команды уже существуют.
5. Для сложной задачи создай ExecPlan по `PLANS.md` и веди его до завершения.
6. Реализуй только текущий этап. Не начинай следующий этап без отдельного задания.
7. В конце выполни self-review и перечисли выполненные команды с результатами.

## Правила декомпозиции

- Одна единица работы должна давать проверяемый вертикальный результат и по возможности укладываться в один логический commit.
- Сначала реализуется минимальный сквозной путь, затем ошибки, безопасность, масштабирование и удобство.
- Большой этап делится по порядку: контракт → тестовый double → backend → frontend → интеграционный тест → эксплуатационные доказательства.
- Не выполняй массовый рефакторинг одновременно с новой функцией.
- Не добавляй инфраструктурную зависимость, если существующий компонент решает задачу без нарушения требований. Любая новая постоянная зависимость требует ADR.
- Не скрывай неопределенность. Для неизвестной внешней системы создай интерфейс, mock и запись в открытых вопросах; не имитируй production-интеграцию.

## Архитектурные инварианты

- Браузер обращается только к frontend и BFF/API портала. Прямые вызовы OpenStack API из JavaScript запрещены.
- OpenStack-токены, application credentials, пароли и сертификаты не передаются в браузер и не сохраняются в `localStorage` или `sessionStorage`.
- Backend повторно проверяет авторизацию для каждого действия. Скрытие кнопки в UI не является средством защиты.
- Keystone и политики сервисов OpenStack остаются окончательным механизмом разрешения OpenStack-операций. Прикладные роли портала могут сужать права, но не расширять их.
- Запрещен прямой доступ к БД Keystone, Nova, Placement, Neutron, Glance, Cinder, Mistral, Watcher и Masakari.
- MariaDB хранит только собственные данные портала, server-side сессии, операции и read model.
- Длительные пользовательские операции выполняются через версионированные и разрешенные workflow Mistral. Worker портала не заменяет Mistral.
- RabbitMQ используется через отдельные учетные данные, vhost, exchange и очереди. Запрещено подключаться к RPC-очередям OpenStack как обычный consumer.
- etcd не используется для бизнес-данных, сессий или очередей. Допустимое использование для leader election требует ADR и доказательства необходимости.
- Должно оставаться два собственных runtime-образа: frontend и backend. API, worker, events и migration запускаются разными командами одного backend-образа.
- Миграции БД выполняются явной одноразовой командой до rolling update. API не запускает миграции автоматически.
- Все list API используют серверную пагинацию, фильтрацию и стабильную сортировку. Запрещено загружать весь inventory в браузер.
- Для больших inventory основной путь чтения идет из read model. События ускоряют обновление, а периодический reconciliation исправляет пропуски.
- Мутирующие запросы поддерживают idempotency key и возвращают `operation_id`.
- Каждая интеграция имеет timeout, ограниченный retry, correlation ID и различает временную и постоянную ошибку.

## Требования безопасности

- Никогда не добавляй в репозиторий реальные `clouds.yaml`, openrc, `.env`, токены, пароли, private key, cookie, дампы БД или production URL с credential.
- Не используй production admin credential. Интеграционные тесты выполняются в отдельном тестовом project с минимальными правами.
- По умолчанию запрещены произвольные shell-команды, Python-код, Jinja-выражения и произвольные имена workflow, поступающие от пользователя.
- Workflow и custom actions выбираются только из серверного allowlist-каталога, имеют JSON Schema входа, версию и список разрешенных ролей.
- Сессия хранится на сервере и представлена в браузере непрозрачной cookie с `HttpOnly`, `Secure` и подходящим `SameSite`. Для изменяющих запросов обязателен CSRF-контроль.
- Idle timeout по умолчанию — 900 секунд. Ограничение числа одновременных сессий конфигурируемо и тестируется.
- Логи и audit events не содержат секретов, токенов, request body с паролями и лишних персональных данных.
- Полный текст внутренней ошибки хранится только в защищенном service log с correlation ID; клиент получает безопасное сообщение.
- Контейнеры запускаются не от root, с минимальными capabilities, read-only filesystem там, где это возможно, и без монтирования container socket.
- Требование ДКБ-69 о полном отсутствии интерпретаторов и shell нельзя считать закрытым для Python backend без утвержденного исключения. Не маскируй этот конфликт.
- Любое изменение авторизации, сессий, workflow, аудита, secrets или deployment требует обновить `docs/11_DKB_TRACEABILITY.md` либо явно указать, почему соответствие не изменилось.

## Стиль реализации

- Идентификаторы кода, имена API, таблиц и событий — на английском.
- Проектная документация и пользовательские тексты PoC — на русском, с готовностью к i18n.
- Backend: строгая типизация, небольшие сервисные слои, зависимости через интерфейсы, отсутствие OpenStack-вызовов в HTTP route handlers.
- Frontend: компоненты не содержат правила авторизации; они получают capabilities от backend.
- Ошибки API имеют единый машинно-читаемый формат.
- Время хранится в UTC; в UI отображается с явным часовым поясом.
- Все схемы БД создаются Alembic-миграциями. Изменение схемы без downgrade/rollback-стратегии запрещено.
- Новые публичные API добавляются только с OpenAPI-схемой и contract tests.

## Ожидаемые команды проекта

По мере появления проекта должны поддерживаться единые команды:

    make bootstrap
    make format
    make lint
    make typecheck
    make test
    make test-integration
    make test-e2e
    make test-load
    make security
    make sbom
    make build
    make up
    make down
    make smoke

Если команда еще не реализована текущим этапом, не заявляй ее успешность. Добавляй ее только вместе с работающей реализацией.

## Definition of Done для любого изменения

Изменение завершено, когда:

- пользовательский результат воспроизводим;
- happy path и ключевые отрицательные сценарии покрыты тестами;
- lint, typecheck и релевантные тесты успешно выполнены;
- схема/API/миграции совместимы с rolling update либо ограничение явно описано;
- не добавлены секреты и небезопасные debug-логи;
- обновлены OpenAPI и проектная документация;
- указаны затронутые ДКБ-требования и доказательства;
- выполнен review diff на регрессии, обход авторизации и раскрытие данных;
- оставшиеся риски перечислены прямо, без заявления ложного соответствия.

## Формат итогового отчета Codex

В завершение каждой задачи сообщи:

1. Какое наблюдаемое поведение появилось.
2. Какие файлы изменены.
3. Какие команды были запущены и каков результат.
4. Какие ДКБ-требования затронуты и какие доказательства созданы.
5. Какие риски, ограничения или внешние зависимости остались.
6. Как безопасно откатить изменение.

Не предлагай автоматически перейти к следующему этапу.
